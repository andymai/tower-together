# Commercial Venues

Families `6`, `10`, and `12` are commercial destinations.

## Included Types

- restaurant = family/type `6`
- retail = family/type `10` (`0x0a`)
- fast food = family/type `12` (`0x0c`)

## Priced Family Row

Retail shop (`family 10`) uses the priced payout table keyed by `rent_level`:

| Tier 0 | Tier 1 | Tier 2 | Tier 3 |
|---:|---:|---:|---:|
| `$20,000` | `$15,000` | `$10,000` | `$4,000` |

Restaurant (`6`) and fast-food (`12`) cashflow still depend on the venue-performance subsystem rather than this simple priced-family row.

## Role

Commercial venues are destination-side facilities. They absorb demand generated by sims from other families.

Generic placement seeds the placed-object record with:

- dirty/display refresh requested
- operational-evaluation latch enabled
- operational score `0xff`
- rent tier `1`
- activation age `0`
- no runtime subtype yet

The family-specific setup then assigns a runtime subtype, allocates sim slots, and creates the
linked venue record:

- restaurant (`6`): `48` sim slots plus one linked `CommercialVenueRecord`
- retail (`10`): `48` sim slots plus one linked `CommercialVenueRecord`
- fast food (`12`): `48` sim slots plus one linked `CommercialVenueRecord`

For parity, the clone should model commercial placement as creating both halves immediately: the
placed object and its linked venue-side record.

## Venue Record

Each venue maintains a sidecar record with, at minimum:

- active capacity limit
- current occupancy / active sim count
- attendance or visitor count
- availability state
- derived readiness/performance state
- per-phase capacity seeds
- negative capacity marker used by dispatch gates

Retail shops also maintain a separate rentable-slot lifecycle on the placed-object side:

- retail recurring-open / recurring-closed state
- operational-evaluation latch
- operational score and activation age
- rent tier
- link to the destination-side venue record

That split matters for parity. Retail "is this shop rented/open?" is not the same field as
the venue record's occupancy/performance state. Restaurants and fast-food venues use the
venue record directly; retail spans both layers.

Availability states:

- `-1`: invalid or stale
- `0`: open
- `1`: partially occupied
- `2`: near-full
- `3`: closed for new customers

## Capacity

Capacity is selected from tier tables and can be affected by:

- star rating
- calendar phase
- progress-override state

Daily capacity recompute:

1. reopen valid venues
2. choose the active capacity seed from:
   - phase A when `calendar_phase_flag == 0`
   - phase B when `calendar_phase_flag != 0`
   - override when `facility_progress_override` is active
3. cap the chosen seed by the venue type's tuning limit
4. floor capacity at `10`
5. write `active_capacity_limit` and its negative gate marker
6. roll `today_visit_count` into `yesterday_visit_count`
7. add the previous day's visit count into the population ledger
8. reset `today_visit_count`, `current_active_count`, and `visitor_count`
9. append the venue to its floor bucket table

Initialization:

- every new venue starts with all three capacity seed bytes set to `10`
- the currently active phase seed is immediately cleared so the next recompute repopulates it
- enabled-link venues start with `active_capacity_limit = 10` and `yesterday_visit_count = 10`
- disabled-link venues start with `active_capacity_limit = 0` and `today_visit_count = 10`

Visible status:

- restaurant and fast-food selected-object status use the normal commercial ordinal; the status
  panel does not special-case their current availability band
- retail is special-cased only for the dormant linked-venue state:
  - linked `availability_state = 0xff` uses the "closed/unrented" retail ordinal
  - every other retail linked state, including off-hours `3`, uses the normal commercial ordinal
- the selected-object status panel therefore keys retail visibility off the linked venue record's
  dormant flag, not off its current occupancy count

Venue bucket tables are maintained separately for retail, restaurant, and fast food and are used by random destination selection.

### Venue Selection Algorithm

Commercial venue customers select a destination via zone-bucketed uniform random selection:

1. the sim's current floor is mapped to one of seven 15-floor zones by `classify_path_bucket_index`: `bucket_index = max(0, (floor - 9) / 15)`, covering floors 5–104
2. one venue record is chosen uniformly at random from all available entries in the matching zone bucket for the requested type
3. if the selected venue is invalid or demolished, the sim receives an immediate retry (delay = 0)

Hotel guests (family 0x21) use a separate path: first pick one of the three commercial types uniformly at random (`rand() % 3`: 0=restaurant, 1=fast food, 2=retail), then select uniformly within that type's zone-0 bucket.

Bucket tables are rebuilt at start-of-day (checkpoint 0).

### Retail Income Timing

Retail shop income is **recurring**, not per-visit or closure-based. `activate_retail_shop_cashflow` registers a continuous cashflow stream via `add_cashflow_from_family_resource(10, rent_level)` that persists while the venue is operational. `deactivate_retail_shop_cashflow` removes it. The priced payout table ($20k/$15k/$10k/$4k by rent_level) reflects the recurring income rate, not a lump-sum payment.

Retail activation is exact:

- first open marks the linked venue record available
- it marks the retail span dirty
- it adds `+10` to the primary family ledger

Retail deactivation is exact:

- it marks the linked venue record dormant
- it clears the operational-evaluation latch and activation age
- it marks the retail span dirty

The binary does **not** use the retail placed-object `unit_status` byte to drive that visible open
/ closed distinction. The selected-object panel checks the linked venue record's dormant byte
instead.

As with offices and condos, the clone does not need to reuse the EXE's record layout, but it
should preserve the separation between the retail rentable slot and the linked venue record.

### Phase A/B/Override Trigger

The active capacity seed column is selected as follows:
- **Override (slot 5):** when `facility_progress_override != 0`
- **Phase A (slot 3):** when `calendar_phase_flag == 0` and no override
- **Phase B (slot 4):** when `calendar_phase_flag != 0` and no override

Override lifecycle each day:
1. Tick 0: `facility_progress_override = 0`, then the periodic override check sets it if `day_counter % 8 == 4` AND tower rank < star-4 AND not already set.
2. Tick 1600: the override is cleared unconditionally.

The override therefore applies only during the first half of every 5th day in each 8-day cycle, while the tower is below 4-star rank.

Per-type tuning:

| Type | Phase A cap | Phase B cap | Override cap | Minimum stay |
|---|---:|---:|---:|---:|
| restaurant | `35` | `50` | `25` | `60` ticks |
| fast food | `35` | `50` | `25` | `60` ticks |
| retail | `25` | `30` | `18` | `60` ticks |

## Income

Commercial income depends on attendance/performance state, not merely on the facility existing.

Closure-time cash mapping:

Restaurant closure payout is keyed by the day's visitor count:

| Visitor count | Cash result |
|---|---:|
| `< 25` | `-$6,000` |
| `25..34` | `$4,000` |
| `35..49` | `$6,000` |
| `>= 50` | `$10,000` |

Fast-food closure payout is keyed by the same thresholds:

| Visitor count | Cash result |
|---|---:|
| `< 25` | `-$3,000` |
| `25..34` | `$2,000` |
| `35..49` | `$3,000` |
| `>= 50` | `$5,000` |

Known derived/performance behavior:

- restaurant and fast-food venues compute a derived state from `visitor_count` using tuning thresholds
- restaurant and fast-food threshold levels are `25`, `35`, and `50`
- restaurant and fast-food both map visitor counts into 4 derived-state slots
- retail always uses derived state `0`
- daily closure accrues income for non-retail commercial types, then derives the visible/performance state from the day's visitor count
- closure moves every live venue to availability state `3`
- restaurant and fast-food income is realized at daily closure, not continuously while occupied

The lowest restaurant and fast-food bands are true losses, not zero-income states.

## Availability

Unavailable venues reject slot acquisition and cause actors to bounce or fall back according to their family logic.

Recovered `CommercialVenueRecord.availability_state` values:

- `0`: available / empty
- `1`: partially occupied (`active_assignment_count == 1..9`)
- `2`: near-full (`active_assignment_count >= 10`)
- `3`: closed for the off-hours sweep; daily closure writes this state and selectors/acquire paths reject it
- `0xff`: dormant / inactive

Slot-acquisition rules:

- invalid or closed venues fail immediately
- venues with more than 39 active occupants return an over-capacity wait result
- first occupant moves the venue to partial state
- the 10th occupant moves the venue to near-full state
- releasing the final occupant reopens the venue
- venue release is blocked until the family-specific minimum-stay timer has elapsed
- slot release resets the sim's saved floor fields back to the venue owner floor on success

Route and queue requirement:

- destination-side visitors only create occupancy and demand effects after a real route is
  accepted through the shared resolver
- over-capacity waits are distinct from no-route failures
- the queueing layer is real: accepted elevator paths create actual carrier queue entries,
  and failed route probes do not

Retail has one extra dispatch gate: when the active capacity limit is `0`, one retail state path refuses to dispatch unless a local aux byte is already set.
